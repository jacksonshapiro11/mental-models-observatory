# Repo Workflow

Canonical rules so local git stays aligned with production.

## Two publish paths (the root cause of mess)

| Path | What it does | Git impact |
|------|----------------|------------|
| **`publish.py` (API)** | Scheduled morning task pushes briefs via GitHub REST API | Updates **remote only**. Local `content/daily-updates/` copies become untracked duplicates. |
| **Local `git push`** | Jackson / agents push code changes | Updates remote from local commits. |

**After API publish:** run `git fetch origin` then either delete matching local brief files (`scripts/repo-hygiene.sh`) or `git pull --rebase`. Never bulk-commit duplicate briefs.

## Daily code changes

1. `git fetch origin`
2. `scripts/repo-hygiene.sh` (optional)
3. `git pull --rebase origin main`
4. Make changes — **never `git add .`** (`.gitignore` blocks `system/` but habits matter)
5. Commit in logical chunks (feat/fix/chore)
6. `git pull --rebase && git push origin main`

If push fails, see `git-push-protocol.md`.

## Brief content

- **Publish:** `publish.py` only (scheduled task or skill). Not local git.
- **Local edits:** only commit if the file differs from `origin/main` (revisions, corrections).
- **factcheck.json:** local QA artifact — gitignored, regenerated by `fact-gate.ts`.
- **Never delete past briefs:** Do not `rm content/daily-updates/*.md`. Do not commit deletions of brief files. Use `scripts/repo-hygiene.sh` after `git fetch` to drop only untracked duplicates. On rebase conflicts, sync from remote: `git restore --source=origin/main content/daily-updates/`.

## Commit hygiene

**Two histories:** `publish.py` commits briefs on remote via API; local git must never duplicate that. That split is intentional — do not try to "sync" brief files into local commits.

| When | What to do |
|------|------------|
| **After morning publish** | `git fetch origin` then `scripts/repo-hygiene.sh` to drop local copies that match remote |
| **Before any code push** | `git pull --rebase origin main` — never push on a stale branch |
| **Brief commits on remote** | Leave as one `brief:` commit per day from the API. Squashing history is optional cleanup, not routine; rebasing/squashing published brief SHAs risks deploy churn |
| **Code commits** | Conventional commits (`feat:`, `fix:`, `chore:`) only — never `git add .` or bulk-commit `content/daily-updates/` |
| **Diverged / duplicate briefs** | Run `repo-hygiene.sh`, then `git pull --rebase`. Do not force-push `main` |

**Rule of thumb:** briefs = API path only; code = local git path only. Mixing them recreates duplicate commits and diverged histories.

## X OAuth

1. Set `TWITTER_CLIENT_ID`, `TWITTER_CLIENT_SECRET`, `SNAPSHOT_SECRET` in Vercel.
2. Visit `/api/x-auth?secret=SNAPSHOT_SECRET` once to store tokens in Redis.
3. Distribute cron reads tokens from Redis via `lib/social/x-oauth.ts`.

## Email

Resend + Redis subscribers (`scripts/send-daily-email.ts`). Beehiiv scripts are legacy — **do not use**. Disable any scheduled Beehiiv draft task in Claude.

**Resend ceiling:** Free tier = 100 emails/day. Upgrade Resend plan when subscriber count approaches **80** (headroom for welcome + daily sends).

**Unsubscribe:** HMAC-signed links in email footer (`UNSUBSCRIBE_SECRET` or `SNAPSHOT_SECRET`). Route: `/api/unsubscribe`.

## Morning publish pipeline

Typical schedule: `brief-morning` ~5:20 AM ET runs `publish.py`, then Vercel deploys.

```
brief-morning (~5:20 AM ET)
  └─ publish.py → GitHub push (full + light .md)
       └─ poll GET /api/publish/health?date=YYYY-MM-DD until lightBrief=true
            (initial 15s delay, then every 10s, timeout 300s)
       └─ THEN POST /api/publish/complete
            ├─ full podcast audio (parallel)
            ├─ super brief audio (parallel)
            ├─ email + X distribute (parallel)
            └─ marketing pack (parallel)
       └─ if health times out: ALERT + do NOT call complete (failsafe cron recovers)

brief-morning-verify (~5:50 AM ET)  ← ADD THIS SCHEDULED TASK if missing
  └─ publish.py --verify --content-dir content/daily-updates
       └─ auto-re-publishes any missing files from local copies

Vercel crons (UTC, EDT ≈ UTC-4):
  9:00 Mon–Fri  /api/dashboard/snapshot
  9:50 daily    /api/publish/backup     — content health check (GitHub + deployed site)
  9:55 daily    /api/publish/complete   — primary idempotent failsafe (~5:55 AM ET)
  10:30 daily   /api/publish/complete   — second retry (~6:30 AM ET); no-op if already done
```

**Why poll, not sleep:** `/api/publish/complete` reads the **deployed Vercel filesystem** via `getBriefLightByDate()`, not GitHub. A fixed 90s wait often raced the deploy (2026-07-08 morning → `skipped: true`). Health poll waits until the live site actually has the light brief.

**Skipped is not success:** complete returns HTTP 409 + `skipped: true` when the brief is missing; publish.py prints `ALERT:` and relies on the failsafe crons (9:55 + 10:30 UTC).

**Residual risk (honest):** Vercel Cron is best-effort, not a literal SLA. Two spaced idempotent retries + health-poll ALERT cover the common deploy-race and single-miss cases; a platform-wide cron outage still needs manual `POST /api/publish/complete`. `publish.py` itself is local (gitignored under `.claude/`) — the morning machine must already have the health-poll script.

### Content backup

- **GET `/api/publish/health`** — `{ date, fullBrief, lightBrief, ready }` from deployed filesystem.
- **POST `/api/publish/backup`** (cron) — if briefs missing on site, checks GitHub API when `GITHUB_TOKEN` is set on Vercel; logs alert with actionable recommendations. **Cannot auto-push from Vercel** — recovery requires `publish.py --verify` on a machine with local brief files.
- **Scheduled task `brief-morning-verify`** — run 30 min after `brief-morning`:

```bash
python .claude/skills/publish-brief/scripts/publish.py --verify \
  --content-dir content/daily-updates
```

### Manual pipeline trigger

```bash
curl -X POST "https://www.cosmictrex.com/api/publish/complete" \
  -H "Authorization: Bearer $CRON_SECRET"
```

Optional backfill: `?date=YYYY-MM-DD`. Step logs in Redis (`distribute:log:{date}`, `audio:log:{date}`, `marketing:pack:{date}`). Full podcast idempotency via `readEpisodeMetadata`.

### Env vars for pipeline

| Var | Used by |
|-----|---------|
| `SNAPSHOT_SECRET` | publish.py → publish/complete; manual curl |
| `CRON_SECRET` | Vercel cron auth |
| `GITHUB_TOKEN` | publish.py; optional on Vercel for `/api/publish/backup` GitHub checks |
