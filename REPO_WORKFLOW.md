# Repo Workflow

Canonical rules so local git stays aligned with production.

## Two publish paths (the root cause of mess)

| Path | What it does | Git impact |
|------|----------------|------------|
| **`publish.py` (API)** | Scheduled morning task pushes briefs via GitHub REST API | Updates **remote only**. Local `content/daily-updates/` copies become untracked duplicates. |
| **Local `git push`** | Jackson / agents push code changes | Updates remote from local commits. |

**After API publish:** run `git fetch origin` then either delete matching local brief files (`scripts/repo-hygiene.sh`) or `git pull --rebase`. Never bulk-commit duplicate briefs.

## Daily code changes

1. `git fetch origin`
2. `scripts/repo-hygiene.sh` (optional)
3. `git pull --rebase origin main`
4. Make changes — **never `git add .`** (`.gitignore` blocks `system/` but habits matter)
5. Commit in logical chunks (feat/fix/chore)
6. `git pull --rebase && git push origin main`

If push fails, see `git-push-protocol.md`.

## Brief content

- **Publish:** `publish.py` only (scheduled task or skill). Not local git.
- **Local edits:** only commit if the file differs from `origin/main` (revisions, corrections).
- **factcheck.json:** local QA artifact — gitignored, regenerated by `fact-gate.ts`.
- **Never delete past briefs:** Do not `rm content/daily-updates/*.md`. Do not commit deletions of brief files. Use `scripts/repo-hygiene.sh` after `git fetch` to drop only untracked duplicates. On rebase conflicts, sync from remote: `git restore --source=origin/main content/daily-updates/`.

## Commit hygiene

**Two histories:** `publish.py` commits briefs on remote via API; local git must never duplicate that. That split is intentional — do not try to "sync" brief files into local commits.

| When | What to do |
|------|------------|
| **After morning publish** | `git fetch origin` then `scripts/repo-hygiene.sh` to drop local copies that match remote |
| **Before any code push** | `git pull --rebase origin main` — never push on a stale branch |
| **Brief commits on remote** | Leave as one `brief:` commit per day from the API. Squashing history is optional cleanup, not routine; rebasing/squashing published brief SHAs risks deploy churn |
| **Code commits** | Conventional commits (`feat:`, `fix:`, `chore:`) only — never `git add .` or bulk-commit `content/daily-updates/` |
| **Diverged / duplicate briefs** | Run `repo-hygiene.sh`, then `git pull --rebase`. Do not force-push `main` |

**Rule of thumb:** briefs = API path only; code = local git path only. Mixing them recreates duplicate commits and diverged histories.

## Email

Resend + Redis subscribers (`scripts/send-daily-email.ts`). Beehiiv scripts are legacy — **do not use**. Disable any scheduled Beehiiv draft task in Claude.

**Resend ceiling:** Free tier = 100 emails/day. Upgrade Resend plan when subscriber count approaches **80** (headroom for welcome + daily sends).

**Unsubscribe:** HMAC-signed links in email footer (`UNSUBSCRIBE_SECRET` or `SNAPSHOT_SECRET`). Route: `/api/unsubscribe`.

## Morning publish pipeline

Typical schedule: `brief-morning` ~5:20 AM ET runs `publish.py`, then Vercel deploys.

```
brief-morning (~5:20 AM ET)
  └─ publish.py → GitHub push (full + light .md)
       └─ poll GET /api/publish/health?date=YYYY-MM-DD until lightBrief=true
            (initial 15s delay, then every 10s, timeout 300s)
       └─ THEN POST /api/publish/complete  (thread is JOINed — not fire-and-forget)
            ├─ full podcast audio (parallel)
            ├─ super brief audio (parallel)
            ├─ email + X distribute (parallel)
            └─ marketing pack (parallel)
       └─ if health times out: ALERT + do NOT call complete (failsafes recover)

FAILSAFE A — GitHub Action (template ready — needs one-time install)
  Weekly-aware template at `docs/ops/publish-complete.workflow.yml`.
  Copy to `.github/workflows/publish-complete.yml` and push with a token that
  has the `workflow` scope; add repo Action secret `CRON_SECRET` or
  `SNAPSHOT_SECRET`. (2026-07-19: code push blocked without that scope.)
  on push to main of content/daily-updates/** (and workflow_dispatch)
  └─ detects daily vs weekly paths
  └─ poll https://www.cosmictrex.com/api/publish/health (custom domain)
       └─ POST /api/publish/complete with Bearer CRON_SECRET or SNAPSHOT_SECRET

FAILSAFE B — Vercel crons (UTC, EDT ≈ UTC-4):
  9:00 Mon–Fri  /api/dashboard/snapshot
  9:50 daily    /api/publish/backup     — content health check
  9:55 daily    /api/publish/complete   — ~5:55 AM ET
  10:30 daily   /api/publish/complete   — ~6:30 AM ET
  11:15 daily   /api/publish/complete   — ~7:15 AM ET (late catch)
  10:15 daily   /api/publish/complete-weekly — morning
  22:00 Sunday  /api/publish/complete-weekly — ~6:00 PM ET (evening catch)
  01:00 Monday  /api/publish/complete-weekly — ~9:00 PM ET Sun (late weekly)
```

### Zoom-out Sunday front page (2026-07-19 / W29)

`lib/weekly-window.ts`: The Weekly wins over a **same-day** daily (`latestDaily > weeklySunday`, not `>=`). Morning daily publish on zoom-out Sunday must still be HELD by the pipeline; this code fix keeps the Weekly on `/daily-update` even if the hold fails.


**Why poll, not sleep:** `/api/publish/complete` reads the **deployed Vercel filesystem** via `getBriefLightByDate()`, not GitHub. A fixed 90s wait often raced the deploy (2026-07-08 morning → `skipped: true`). Health poll waits until the live site actually has the light brief. complete itself also waits up to 90s before returning 409.

**Skipped is not success:** complete returns HTTP 409 + `skipped: true` when the brief is missing after wait; publish.py prints `ALERT:`.

### Critical: Vercel Cron + Deployment Protection (Jul 10 2026)

Vercel Cron invokes the **deployment `*.vercel.app` host**, not the custom domain. With SSO Protection set to `all_except_custom_domains`, those requests get **HTTP 302 → Vercel login**. Cron jobs **do not follow redirects**, so every failsafe silently no-ops while looking "successful" in the dashboard.

**Required setting:** Project → Deployment Protection → Vercel Authentication → **Preview only** (`ssoProtection.deploymentType = "preview"`). Custom domain stays public; production `*.vercel.app` must reach the function (401 from our auth is fine; 302 is fatal).

**Auth:** `lib/cron-auth.ts` accepts `x-vercel-cron` / `x-vercel-cron-schedule` / `User-Agent: vercel-cron`, Bearer `CRON_SECRET`/`SNAPSHOT_SECRET`, or `?secret=`.

**Residual risk (honest):** Vercel Cron remains best-effort. GHA on content push is the reliable event-driven backup (hits cosmictrex.com). `publish.py` is local (gitignored under `.claude/`) — morning machine must have the join+health-poll script. Repo must have `CRON_SECRET` or `SNAPSHOT_SECRET` in GitHub Actions secrets for the workflow.

### Content backup

- **GET `/api/publish/health`** — `{ date, fullBrief, lightBrief, ready }` from deployed filesystem.
- **POST `/api/publish/backup`** (cron) — if briefs missing on site, checks GitHub API when `GITHUB_TOKEN` is set on Vercel; logs alert with actionable recommendations. **Cannot auto-push from Vercel** — recovery requires `publish.py --verify` on a machine with local brief files.
- **Scheduled task `brief-morning-verify`** — run 30 min after `brief-morning`:

```bash
python .claude/skills/publish-brief/scripts/publish.py --verify \
  --content-dir content/daily-updates
```

### Manual pipeline trigger

```bash
curl -X POST "https://www.cosmictrex.com/api/publish/complete" \
  -H "Authorization: Bearer $CRON_SECRET"
```

Optional backfill: `?date=YYYY-MM-DD`. Weekly: `?weekly=YYYY-Www` (or `/api/publish/complete-weekly?weekly=YYYY-Www`).

```bash
# Weekly audio + email recovery (e.g. W28 stranded 2026-07-12)
curl -X POST "https://www.cosmictrex.com/api/publish/complete?weekly=2026-W28" \
  -H "Authorization: Bearer $CRON_SECRET"
```

Weekly health poll: `GET /api/publish/health?weekly=YYYY-Www` → `{ lightBrief, fullBrief, ready }`.
`scripts/publish-weekly.py` now health-polls then POSTs complete (same contract as daily `publish.py`).
GHA `publish-complete.yml` detects `content/daily-updates/weekly/` pushes and calls `complete?weekly=`.

Step logs in Redis (`distribute:log:{date}`, `audio:log:{date}`, `marketing:pack:{date}`). Full podcast idempotency via `readEpisodeMetadata`.
